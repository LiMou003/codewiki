## 2. 预处理模块

### 2.1 模块职责

预处理模块（`api/data_pipeline.py`）负责将原始代码仓库转化为可供向量化处理的结构化文档集合，是整个知识库构建流程的起点。其核心职责包括：

- 克隆远程代码仓库（GitHub / GitLab / Bitbucket / 企业自建 Git）
- 对文件系统进行扫描与过滤（排除目录、包含目录、文件名白/黑名单）
- Token 数量估算与超限文件跳过
- 构建结构化的 `Document` 对象供下游模块使用

### 2.2 仓库克隆原理

系统通过调用 `git clone --depth=1 --single-branch` 进行**浅克隆**（Shallow Clone），只获取最新一个提交的快照，大幅缩短克隆时间和磁盘占用。

对于私有仓库，系统将 Personal Access Token 嵌入 URL（`https://{token}@{host}/owner/repo`），并使用 `urllib.parse.quote` 对 Token 进行 URL 编码，防止特殊字符破坏 URL 结构。错误日志中会将 Token 替换为 `***TOKEN***` 以防止泄露。

克隆后的仓库存储于用户家目录下的 `~/.adalflow/repos/{owner}_{repo_name}/`，下次访问相同仓库时直接复用，无需重复克隆。

### 2.3 文件扫描与过滤

系统支持两种过滤模式：

**排除模式（Exclusion Mode）：**
- 默认排除 `node_modules`、`.git`、`__pycache__`、`dist`、`build` 等通用目录
- 通过配置文件 `api/config/repo.json` 可扩展排除规则
- 用户可在前端通过 UI 动态指定额外的排除目录或文件

**包含模式（Inclusion Mode）：**
- 当用户指定 `included_dirs` 或 `included_files` 时进入此模式
- 仅处理白名单目录/文件中的内容，适用于聚焦某个子模块的场景

两种模式互斥：只要指定了任意包含规则，系统自动切换为包含模式，忽略所有排除规则。

### 2.4 文件类型分类

```
代码文件（is_code = True）：
  .py .js .ts .jsx .tsx .java .cpp .c .h .go .rs .php .swift .cs ...

文档文件（is_code = False）：
  .md .txt .rst .json .yaml .yml .xml ...
```

代码文件额外标注 `is_implementation`（通过启发式规则判断是否为实现文件而非测试文件），供下游模块在检索时提升实现代码的优先级。

### 2.5 Token 数量估算

在将文件内容纳入文档集合之前，系统使用 `tiktoken` 库对文件内容进行 Token 计数，并设置上限阈值（`MAX_EMBEDDING_TOKENS = 8192`）：

- 代码文件：若 Token 数 > 8192 × 10，则跳过（超大生成文件，如压缩 bundle）
- 文档文件：若 Token 数 > 8192 × 1，则跳过（防止单文件超过模型限制）

对于非 OpenAI 的 Embedding 提供商（Google、Bedrock、DashScope、Ollama），Token 估算使用通用的 `cl100k_base` 编码；OpenAI 则使用 `text-embedding-3-small` 专用编码。

## 3. 基于 Tree-sitter 的多语言函数级代码切分模块

### 3.1 设计动机

朴素的按字符数切分（Fixed-Length Chunking）会将函数或类从中间截断，破坏代码语义完整性，严重影响检索质量。为此，本系统采用 **Tree-sitter AST（抽象语法树）解析**，在语义边界处精准切分代码——每个切块对应一个完整的函数、方法或类定义。

### 3.2 Tree-sitter 工作原理

Tree-sitter 是一个**增量式、健壮、多语言**的解析器生成工具和运行时库，由 GitHub 开发并开源。其核心特点：

1. **原生多语言支持：** 每种语言有独立的语法模块（如 `tree-sitter-python`、`tree-sitter-java`），以共享库形式加载
2. **具体语法树（CST）：** 解析结果是完整的具体语法树，保留所有标点和空白，节点有精确的字节偏移和行列坐标
3. **容错解析：** 即使代码有语法错误，Tree-sitter 也能输出尽可能合理的树结构，不会整体失败
4. **高性能：** 使用 C 语言编写，解析速度极快，支持实时增量更新（编辑器场景）

在本项目中，Tree-sitter 以 Python 绑定形式调用（`tree_sitter` 包），流程为：
1. 将 Python 字符串以 UTF-8 编码为字节序列
2. 调用语言特定的 `Parser.parse(bytes)` 方法，返回语法树对象
3. 遍历树的节点，根据节点类型（`function_definition`、`class_declaration` 等）提取目标代码块

### 3.3 支持的编程语言及 AST 节点映射

| 语言        | 文件扩展名              | 提取的节点类型                                              |
|-------------|------------------------|-----------------------------------------------------------|
| Python      | `.py`                  | `function_definition`（函数）、`class_definition`（类）     |
| JavaScript  | `.js`, `.jsx`          | `function_declaration`、`class_declaration`、`method_definition` |
| TypeScript  | `.ts`                  | 同 JavaScript                                              |
| TSX         | `.tsx`                 | 同 JavaScript                                              |
| Java        | `.java`                | `method_declaration`、`constructor_declaration`、`class_declaration`、`interface_declaration`、`enum_declaration` |
| Go          | `.go`                  | `function_declaration`（函数）、`method_declaration`（方法） |
| Rust        | `.rs`                  | `function_item`（函数）、`impl_item`（实现块）               |
| C / C++     | `.c`, `.cpp`, `.cc`    | `function_definition`（函数）                               |
| C#          | `.cs`                  | `method_declaration`、`constructor_declaration`、`class_declaration`、`interface_declaration` |

### 3.4 函数名称提取机制

从 AST 节点中提取函数/方法名是实现精准元数据的关键步骤，不同语言的节点结构差异较大：

- **通用策略：** 遍历节点的直接子节点，寻找类型为 `identifier`、`property_identifier`、`type_identifier` 的子节点，取其文本作为名称
- **C/C++ 特例：** 函数名深藏于 `function_declarator` 子节点内，需两级遍历；若存在指针声明（`pointer_declarator`），还需进一步向下查找
- **Go 方法特例：** `method_declaration` 中第一个 `identifier` 属于接收者类型，方法名实为 `field_identifier` 类型的子节点

### 3.5 父类名称提取（上下文关联）

为了让每个方法切块携带所属类的信息，系统在 AST 中向上遍历父节点链，查找 `class_definition`、`class_declaration`、`impl_item` 等类型的祖先节点，并提取其名称作为 `class_name` 字段，实现"方法→所属类"的上下文关联。

### 3.6 切块粒度控制与大小限制

对于一个包含大量方法的超大类（如数百行的工具类），其整个类节点的文本可能超过 Embedding 模型的上下文长度限制。系统的处理策略：

1. **类和 impl 块的递归遍历：** 遇到类节点时，先生成类级别切块，再递归提取其内部的方法定义为独立切块——即同一段代码会以不同粒度存储，兼顾全局语义和局部细节
2. **最大切块尺寸限制：** 单个切块的字符数不超过 `chunk_size × 3`（默认 `1500 × 3 = 4500` 字符），超出部分截断，防止 Embedding 调用失败

### 3.7 固定长度兜底策略

对于 Tree-sitter 不支持的语言（如 Shell、Dockerfile、SQL）或解析失败的场景，系统回退到**基于行边界的固定长度切分**：

- 目标切块大小：1500 字符
- 相邻切块重叠：200 字符（按行对齐，不截断单行）
- 切块类型标注为 `fixed_length`

### 3.8 切块数据结构

每个切块封装为 `CodeChunk` 数据类：

```
CodeChunk:
  text          : 切块的原始代码文本
  file_path     : 源文件相对路径
  language      : 编程语言标识符（如 "python", "java"）
  chunk_type    : 切块类型（"function" / "class" / "method" / "impl" / "fixed_length"）
  function_name : 函数或方法名（可选）
  class_name    : 所属类名（可选）
  start_line    : 起始行号（1-based）
  end_line      : 结束行号（1-based）
  metadata      : 附加元数据（如原始 AST 节点类型）
```

### 3.9 处理流程图

```
Document (原始源码)
        │
        ▼
  确定文件语言
  (扩展名 → 语言 ID)
        │
   ┌────┴────┐
   │         │
  code     非 code
   │         │
   ▼         ▼
加载 Tree-sitter   固定长度切分
```

## 4. 向量化与知识库构建模块

### 4.1 技术架构概述

向量化模块将 `CodeChunk` 文本转换为高维稠密向量（Embedding），并持久化存储到 Qdrant 向量数据库，构成可被语义检索的代码知识库。整个流程由 `DatabaseManager` 类统一编排，对外暴露 `prepare_database()` 接口。

### 4.2 Embedding 技术原理

**文本嵌入（Text Embedding）** 是将自然语言或代码文本映射到高维实数向量空间的技术。语义相似的文本在向量空间中距离较近。通过计算查询向量与知识库中存储向量的余弦相似度，可以实现语义级别的精准检索，而无需依赖关键词精确匹配。

系统支持多家 Embedding 提供商，通过配置文件 `api/config/embedder.json` 动态切换：

| 提供商      | 代表模型                            | 向量维度 | 适用场景              |
|-------------|-------------------------------------|----------|-----------------------|
| OpenAI      | `text-embedding-3-small`            | 256–1536 | 通用首选，性价比高    |
| Google      | `gemini-embedding-001`              | 可变     | Google Cloud 生态     |
| DashScope   | `qwen3-embedding` (Qwen 系列)       | 可变     | 国内访问，阿里云生态  |
| AWS Bedrock | `amazon.titan-embed-text-v2:0`      | 1024     | AWS 生态              |
| Ollama      | 本地模型（如 `nomic-embed-text`）   | 可变     | 离线/隐私场景         |

**批量处理优化：** 非 Ollama 提供商支持批量嵌入（batch_size 默认 500），大幅降低 API 调用次数和延迟。Ollama 因本地模型限制采用逐条处理模式。

### 4.3 Qdrant 向量数据库

**Qdrant** 是一个高性能开源向量数据库，专为近似近邻（ANN）搜索设计，核心特点：

- **Collection（集合）：** 类似关系型数据库的"表"，每个仓库对应一个独立集合，命名规则为 `codewiki_{owner}_{repo_name}`
- **Point（点）：** 集合中的基本存储单元，每个点包含：UUID 主键、浮点向量、任意 JSON Payload（存储代码元数据）
- **余弦距离：** 向量相似度度量采用余弦相似度（Cosine Similarity），对向量长度不敏感，适合文本/代码语义匹配
- **部署灵活：** 支持纯内存模式（开发/测试）和远程持久化服务（生产，默认端口 6333）

每个存储 Point 的 Payload 结构：

```
{
  "text"          : "<代码切块原文>",
  "file_path"     : "src/utils/parser.py",
  "function_name" : "parse_expression",
  "class_name"    : "ExpressionParser",
  "chunk_type"    : "method",
  "language"      : "python",
  "start_line"    : 42,
  "end_line"      : 78,
  "is_code"       : true,
  "is_implementation": true
}
```

### 4.4 知识库构建完整流程

```
DatabaseManager.prepare_database(repo_url)
        │
        ▼
   _create_repo()              下载仓库，确定存储路径
        │
        ▼
   collection_exists_for_repo? ──── 是 ──→ 跳过重建，直接复用（增量场景）
        │ 否
        ▼
   创建 Qdrant Collection       余弦距离，向量维度由 Embedder 决定
        │
        ▼
   read_all_documents()         递归扫描，过滤，读取文件，构建 Document 列表
        │
        ▼
   split_and_index_to_qdrant()
     │
     ├─ 对每个 Document：
     │    ├─ is_code=True  → TreeSitterCodeSplitter.split_code()  → CodeChunk[]
     │    └─ is_code=False → split_text_fixed()                   → CodeChunk[]
     │
     ├─ 对每个 CodeChunk：
     │    └─ _get_embedding_vector(embedder, chunk.text) → [float]
     │

              │

        │

```

### 4.5 嵌入向量一致性校验

不同 API 调用失败或模型切换可能导致同一批文档产生不同维度的向量（如部分成功为 256 维、部分失败后回退为其他维度）。Qdrant 要求同一集合内所有向量维度一致，否则抛出异常。

系统通过 `_validate_and_filter_embeddings()` 进行校验：统计所有文档的嵌入向量维度分布，取出现频次最高的维度作为目标维度，过滤掉维度不符的文档，并记录警告日志，确保整批写入不会因个别异常中断。

## 5. 检索问答模块（RAG）

### 5.1 RAG 技术原理

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与生成式 AI 相结合的范式，解决了纯 LLM 在专有领域（如私有代码仓库）知识不足的问题：

```
用户问题
   │
   ▼
Embedding 模型 → 问题向量
   │
   ▼
向量数据库检索 → Top-K 相关代码切块
   │
   ▼
构建上下文提示词（System Prompt + 历史对话 + 相关切块 + 用户问题）
   │
   ▼
LLM 生成回答
   │
   ▼
流式返回给用户
```

**核心优势：**
- LLM 无需事先"记住"所有代码，仅在推理时动态注入最相关的片段
- 支持超大代码库（理论上无上限），LLM 只处理检索到的少量上下文
- 可精确溯源——每个回答背后有具体的文件路径和行号支持

### 5.2 检索流程

**向量检索（`_qdrant_search`）：**

1. 调用配置的 Embedding 模型，将用户查询转换为查询向量
2. 向 Qdrant 发起 ANN（近似近邻）搜索，默认返回 Top-20 个相似切块
3. 将命中结果封装为 `RetrieverOutput` 对象，每个文档携带完整元数据（文件路径、函数名、行号等）

**结果后处理：**

- 按文件路径分组，将同一文件的多个切块合并展示，减少上下文重复
- 构建结构化上下文字符串，以 `## File Path: xxx` 为标题分隔不同文件

### 5.3 提示词工程（Prompt Engineering）

系统使用 Jinja2 模板化的多部分提示词（`RAG_TEMPLATE`）：

```
<START_OF_SYS_PROMPT>
  [系统角色定义]
  [输出格式要求]
<END_OF_SYS_PROMPT>

<START_OF_CONVERSATION_HISTORY>   ← 注入历史对话（多轮推理）
  Turn 1: User / Assistant
  Turn 2: User / Assistant
  ...
<END_OF_CONVERSATION_HISTORY>

<START_OF_CONTEXT>                ← 注入检索到的代码切块
  1. File Path: xxx
     Content: ...
  2. File Path: yyy
     Content: ...
<END_OF_CONTEXT>

<START_OF_USER_PROMPT>            ← 用户当前问题
  [用户查询]
<END_OF_USER_PROMPT>
```

**系统角色提示设计原则：**
- 明确语言检测：自动识别用户查询的语言（中文/英文），并以相同语言回答
- 格式约束：要求使用 Markdown 格式，禁止在回答最外层使用 ` ``` ` 代码围栏
- 精确性约束：严格基于检索到的代码/文档回答，不允许猜测或编造引用

### 5.4 输出解析

系统使用 AdalFlow 框架的 `DataClassParser` 对 LLM 的结构化输出进行解析：

```python
class RAGAnswer:
    rationale : str  # 推理过程（Chain-of-Thought）
    answer    : str  # 最终答案（Markdown 格式）
```

LLM 被要求以 JSON 格式输出这两个字段，`DataClassParser` 提取 `answer` 字段作为最终回复，丢弃中间推理过程，避免向用户暴露内部思考链。

## 6. 多轮推理模块

### 6.1 对话历史管理

为支持连续追问（如"请详细解释其中的异步处理部分"），系统实现了会话级对话历史管理。

**核心数据结构：**

```
CustomConversation
  └── dialog_turns: List[DialogTurn]
          │
          └── DialogTurn
                ├── id           : UUID（唯一标识符）
                ├── user_query   : UserQuery(query_str)
                └── assistant_response : AssistantResponse(response_str)
```

**`Memory` 组件** 封装了对话历史的增删查操作：
- `call()` 方法返回以 turn ID 为键的 `OrderedDict`，供模板渲染使用
- `add_dialog_turn()` 负责安全地追加新对话轮次，内置异常恢复机制

每次新请求到来时，系统将历史消息列表中前 N-1 条（已完成的对话轮次）重新导入 Memory，构建完整的对话上下文，传递给 RAG 模板渲染。

### 6.2 深度研究模式（Deep Research）

深度研究模式是多轮推理的高级应用，通过最多 **5 轮迭代**对同一问题进行递进式深入分析：

| 迭代轮次   | 任务                                                  | 系统提示词策略                               |
|-----------|-------------------------------------------------------|---------------------------------------------|
| 第 1 轮    | 制定研究计划，给出初步发现，明确研究方向              | 要求以 `## Research Plan` 开头，`## Next Steps` 结尾 |
| 第 2–4 轮  | 逐步深入探索，补充新发现，不重复已覆盖内容            | 要求以 `## Research Update N` 开头          |
| 第 5 轮    | 综合所有迭代的发现，给出完整结论                      | 要求以 `## Final Conclusion` 开头           |

**触发条件：** 用户消息中包含 `[DEEP RESEARCH]` 标签时自动激活，系统从消息中移除该标签，计算当前轮次，选择对应的系统提示词模板。

**设计优势：** 单次提问可能因上下文窗口限制无法充分分析大型代码库；迭代式研究通过逐轮聚焦不同侧面，最终综合得出全面结论，有效规避了上下文限制问题。

### 6.3 对话历史的数据库持久化

用户的完整对话历史持久化于关系型数据库（MySQL/SQLite），通过 `Conversation` 和 `ConversationMessage` ORM 模型管理：

```
users (用户)
  └── conversations (对话会话)
            ├── repo_owner / repo_name / repo_type  ← 关联仓库
            ├── title                                ← 对话标题
            └── conversation_messages (消息)
                      ├── role: "user" | "assistant"
                      ├── content: 消息正文
                      └── token_count: Token 数量（可选）
```

这使得用户可以跨会话查看历史问答，也为未来实现多设备同步提供了基础。

## 7. 基于 WebSocket 的流式输出

### 7.1 为什么使用 WebSocket

对于大语言模型的推理输出，传统 HTTP 请求/响应模型存在明显缺陷：用户必须等待 LLM 生成完整回答后才能收到响应，对于长回答可能需要等待 30–60 秒，体验极差。

**WebSocket** 提供了全双工、持久化的 TCP 通信通道，允许服务器主动向客户端推送数据：
- LLM 每生成一个 token/文本片段，立即通过 WebSocket 发送给前端
- 前端实时渲染，用户看到文字"逐渐出现"的流式效果
- 连接保持活跃直到本次回答完成，双方均可主动关闭

### 7.2 流式输出实现架构

**后端（FastAPI WebSocket 端点）：**

```
客户端连接 WebSocket
      │
      ▼
接收 JSON 请求数据（ChatCompletionRequest）
      │
      ▼
初始化 RAG 实例，prepare_retriever()
      │
      ▼
重建历史对话（导入前 N-1 轮对话到 Memory）
      │
      ▼
向量检索，获取相关代码切块
      │
      ▼
构建完整提示词（系统提示 + 历史 + 上下文 + 问题）
      │
      ▼
调用 LLM（stream=True）           ←── 关键：启用流式模式
      │
      ├── Ollama：      async for chunk in model.astream()
      ├── OpenAI：      async for event in response
      ├── OpenRouter：  async for chunk in response
      ├── Azure AI：    async for event in response
      ├── DashScope：   async for text in client.acall(stream=True)
      └── Google：      response = model.generate_content(stream=True)
                        for chunk in response: ...
      │
      ▼
await websocket.send_text(chunk_text)   ← 每个片段立即推送
      │
      ▼
回答生成完毕，存入 Memory（对话历史）
      │
      ▼
WebSocket 连接关闭（或等待下一个问题）
```

### 7.3 超大输入处理策略

当用户输入的消息超过 8000 个 token 时（通过 tiktoken 预估），系统会：

1. 跳过 RAG 检索步骤，直接将用户问题传递给 LLM（避免上下文叠加超限）
2. 如果整个提示词仍超出模型上下文窗口，启用**简化提示词兜底策略**：去除历史对话和检索上下文，仅保留系统提示和用户当前问题，确保至少返回一个有意义的回答

### 7.4 前端 WebSocket 接入

前端（Next.js）通过浏览器原生 WebSocket API 建立连接：

1. 用户点击发送后，打开 WebSocket 连接至 `/ws/chat`
2. 发送序列化为 JSON 的 `ChatCompletionRequest` 对象
3. 监听 `onmessage` 事件，将收到的文本片段追加到当前消息气泡
4. 使用 `react-markdown` 实时渲染接收到的 Markdown 文本
5. 收到连接关闭或错误信号后，标记消息为"已完成"状态

### 7.5 错误处理

WebSocket 会话中出现的各类错误均通过 `send_text` 以文本形式返回给前端（前缀为 `Error:`），而非直接关闭连接。这使得前端可以友好地展示错误信息，而不是显示空白或连接失败提示。

## 8. 用户认证模块

### 8.1 技术选型

| 组件         | 选型                                        | 说明                                          |
|--------------|---------------------------------------------|-----------------------------------------------|
| 密码哈希     | PBKDF2-SHA256（`passlib`）                  | 安全性优于 bcrypt，无 72 字节长度限制          |
| 令牌方案     | JWT（HS256 签名，`python-jose`）            | 无状态，便于横向扩展                           |
| ORM          | SQLAlchemy 异步（`asyncio` + `aiosqlite`/`aiomysql`） | 支持 SQLite（开发）和 MySQL（生产）双模式 |
| HTTP 路由    | FastAPI `APIRouter`（前缀 `/auth`）         | 声明式路由，自动生成 OpenAPI 文档              |

### 8.2 数据库模型

用户认证相关的 ORM 模型定义于 `api/database/models.py`：

**`users` 表核心字段：**

```
id            : String(36)  — UUID 主键
username      : String(64)  — 用户名（全局唯一）
email         : String(255) — 电子邮件（全局唯一）
password_hash : String(255) — 密码哈希值（PBKDF2-SHA256）
is_active     : Boolean     — 账户是否启用
created_at    : DateTime    — 账户创建时间
```

关联表：`user_settings`（用户偏好，一对一）、`conversations`（对话历史，一对多）。

### 8.3 注册流程

**接口：** `POST /auth/register`

```
客户端 → POST /auth/register
         { username, email, password }
              │
              ▼
   检查用户名/邮箱唯一性（数据库查询）
              │
         ┌────┴────┐
         │ 冲突    │ 无冲突
         ▼         ▼
   409 Conflict  PBKDF2-SHA256 哈希密码
                      │
                      ▼
                 写入 users 表
                      │
                      ▼
                 签发 JWT Access Token
                 (sub=user_id, exp=+60min)
                      │
                      ▼
              返回 TokenResponse
              { access_token, token_type, user }
```

**安全要求：**
- 密码最短 8 个字符（Pydantic `Field(min_length=8)` 验证）
- 密码以 PBKDF2-SHA256 哈希后存储，禁止明文
- 用户名最短 3 个字符，最长 64 个字符

### 8.4 登录流程

**接口：** `POST /auth/login`

```
客户端 → POST /auth/login
         { username, password }
              │
              ▼
   按用户名查询用户
              │
         ┌────┴────┐
         │ 不存在  │ 存在
         ▼         ▼
   401 Unauthorized  _verify_password(plain, hash)
                          │
                     ┌────┴────┐
                     │ 验证失败 │ 验证成功
                     ▼         ▼
              401 Unauthorized  检查 is_active
                                    │
                               ┌────┴────┐
                               │ 停用    │ 正常
                               ▼         ▼
                        403 Forbidden  签发 JWT
                                           │
                                           ▼
                                    返回 TokenResponse
```

注意：用户不存在和密码错误返回**相同的 401 错误信息**（"用户名或密码错误"），防止用户枚举攻击（Username Enumeration）。

### 8.5 JWT 令牌机制

**JWT（JSON Web Token）** 是一种开放标准（RFC 7519），将认证信息编码到自包含的令牌中，无需服务器端会话存储。

本系统 JWT Payload 结构：

```json
{
  "sub": "user-uuid",            // 用户唯一标识
  "iat": 1710000000,             // 签发时间（Unix timestamp）
  "exp": 1710003600              // 过期时间（默认 60 分钟后）
}
```

令牌使用 **HMAC-SHA256（HS256）** 算法对 Header+Payload 进行签名，密钥通过环境变量 `JWT_SECRET_KEY` 配置（生产环境应使用高熵随机字符串，长度建议 ≥ 32 字节）。

**令牌验证（`GET /auth/me`）：**

1. 从 HTTP `Authorization: Bearer <token>` 请求头提取令牌
2. 验证签名完整性（防篡改）
3. 验证 `exp` 字段未过期
4. 提取 `sub` 字段，查询数据库确认用户存在且处于激活状态
5. 返回用户 Profile 信息

### 8.6 认证中间件集成

FastAPI 通过 `Depends(_get_current_user)` 实现声明式依赖注入，任何需要认证的路由只需在参数列表中声明该依赖，FastAPI 自动执行令牌验证流程，无需在每个路由函数中重复编写认证逻辑。

### 9.1 各模块技术要点汇总

| 模块               | 核心技术                               | 关键创新点                                     |
|--------------------|----------------------------------------|------------------------------------------------|
| 预处理             | git clone、tiktoken、glob              | 浅克隆加速、Token 超限跳过、双模过滤            |

| 多轮推理           | 对话历史管理 + 深度研究迭代            | 最多 5 轮渐进式研究、角色提示词随轮次动态调整   |
